## Сетёвка: какие нужно решать проблемы

- Сетевой слот
	-> Хранит сетевой айдишник, эпоху, ссылку на объект (если есть), список доменов.
	
	Эпоха увеличивается после каждого удаления объекта. Разница эпох позволяет отловить передачу данных для новой/старой версии объекта.
	Переполнение эпохи допустимо.

	Если отправлять конструкторы перед другими доменами, то несовпадение эпохи может быть: 
	* признаком переупорядовачивания сообщений,
	* неполученным (неотправленным) деструктором - можно игнорировать, если получен конструктор с эпохой на единицу больше,
	* багом сетёвки. 

- Деление сетевых объектов на домены
	-> Сетевые объекты раскладываются на сетевые домены - части, которые передаются по сети самостоятельно.
	Сетевой домен обрабатывает отправку и получение данных (если есть),
	указывает получателей (для SV->CL) и допустимых отправителей (для CL->SV).
	Сетевой домен отвечает за переотправку своих данных новым игрокам.

- Инициализация объекта
	-> Объект имеет домен переменной 0 (конструктор/деструктор).
	Когда передаются пустые данные, это деструктор, когда непустые - конструктор.

	На сервере после инициализации объекта (или после вызова функции :Net_Initialized() для статических объектов) включается домен 0 с данными для конструктора. Данные включают в себя тип объекта и пользовательские параметры конструктора.
	Домен 0 отправляет данные, кому надо.

	Клиенты получают сообщение для домена 0 для слота с объектом или без него. Если размер данных != 0, это конструктор.
	Если слот имеет объект, объект удаляется.
	Из сообщения читается тип объекта, создаётся неинициализированный объект этого типа.
	Вызывается :Net_Configure(), что создаёт домены.
	Домен 0 читает данные.
	Вызывается :Init() и :Net_Initialized().

	- Порядок инициализации на клиенте
		-> Инициализировать в порядке получения сообщений конструктора. Зависимость между объектами (напр. необходимость иметь инициализированным другой объект для полной инициализации этого) решается через второю фазу инициализации и каллбек с семафором.


- Передача нормальных данных
	-> Данные содержатся в домене N != 0.
	
	На отправляющей стороне что-то инициирует отправку (изменение данных либо появление нового получателя либо следующая фаза передачи данных).
	Домен определяет список получателей (Domain:GetRecipients)
	Создаётся сообщение.
	Домен пишет данные в сообщение (Domain:WriteMessage).
	Сообщение отправляется.

	На получающей стороне получают сообщение домена N какого-то слота.
	Если слот неиницилизирован, см. "Клиент получил данные для объекта неизвестного типа, что делать".
	Если слот имеет объект, получаем объект домена.
	Домен проверяет фильтр получения (Domain:FilterRecieve), если игнорируем, то игнорируем.
	Домен читает данные из сообщения (Domain:ReadMessage)

	- Клиент получил данные для объекта неизвестного типа, что делать?
		-> На клиенте: создать слот, поместить сырые данные в данные домена, когда будет создан объект, передать эти данные домену.
		На сервере: игнорировать.

- Удаление объекта
	-> Объект имеет домен переменной 0 (конструктор/деструктор).
	Когда передаются пустые данные, это деструктор, когда непустые - конструктор.

	На сервере удаление вызывается функцией :Remove объекта или пересозданием статического объекта (ловится повторным :Net_Initialized). 
	Всем, кто знают что-либо об объекте, отправляется пустое сообщение домена 0.
	Затем обнуляется слот и эпоха увеличивается на 1.

	На клиенте удаление объекта вызывается получением пустого сообщения домена 0 или вызовом :Remove или пересозданием статического объекта.
	Случай повторного удаления статического объекта (один раз через hot reload на клиенте, один раз через сетёвку) отлавливается совпадением эпох.
	При удалении объекта обнуляется слот и эпоха увеличивается на 1.

	- Правильная обработка случаев, когда объект был удалён, был создан новый объект с тем же айдишником, 
	и затем пришли данные для старого объекта
		-> Движковая сетёвка, ВРОДЕ БЫ, не может прислать старые данные после новых.
		Т.е. единственный способ, как такое может получиться, это из-за багов моей сетёвки.
		-> Можно ожидать, что в нормальных условиях увеличение эпохи на +1..+10 - это более новый объект, а всё остальное - wraparound от более старых.
	
- Сетёвка для статических классов
	-> Объект готов к сетёвке не по завершению :OnInit, а по вызову :Net_Initialized.
	Объект сбрасывается при повторном вызове :Net_Initialized (отловит hot reload)

- Фильтрация отправки

- Откладывание второй фазы инициализации до доступности необходимых объектов-ссылок
	-> Фактически, я реализую каллбек с семафором (счёткиком невыполненных предусловий).
	function Obj:NetSetup()
		local asyncinit = CR.SemaphoredCallback()
		self._asyncinit = asyncinit
		
		-- [setup domains]
		
		if CLIENT then
			asyncinit:WaitFor(self._dependencySlot1)
			asyncinit:WaitFor(self._dependencySlot2)
		end
	end
	
	function Obj:OnInit(...)
		self.Base.OnInit(self, ...)
		
		self._asyncinit:DoOnce(function() self:AsyncInit() end) -- Вызывает каллбек сразу, если семафор в нуле
	end
	
	function Obj:AsyncInit()
		local dep1 = self._dependencySlot1.Obj
		local dep2 = self._dependencySlot2.Obj
		
		assert(IsValid(dep1) and IsValid(dep2))
	end

function Obj:Net_Setup()
	-- Transmitted from server to all clients allowed by the recieve filter.
	-- Transmitted on update (:MarkUpdated()) or to newly-connected allowed players.
	self._varDomain = self:Net_AddDomain(CR.Class.NetVar({
		Send = function() -- SERVER
			net.WriteVector(self.Pos)
			net.WriteUInt(self.Var1, 32)
			net.WriteString(self.Var2)
			net.WriteUInt(#self.Array, 16)
			for _, v in ipairs(self.Array) do
				net.WriteUInt(v.Num, 4)
				net.WriteVector(v.Pos)
			end
		end,
		Recv = function(buf, len) -- CLIENT
			self.Pos = buf:ReadVector()
			self.Var1 = buf:ReadUInt(32)
			self.Var2 = buf:ReadString()
			
			local arr = {}
			for i = 1, buf:ReadUInt(16) do
				arr[i] = {
					Num = buf:ReadUInt(4),
					Pos = buf:ReadVector()
				}
			end
			
			self.Array = arr
		end,
		Filter = 
	}))
end

function Obj:Change()
	self.Var1 = self.Var1 + 1
	self._varDomain:MarkUpdated()
end
