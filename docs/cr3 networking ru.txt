## Сетёвка: какие нужно решать проблемы

- Сетевой слот
	-> Хранит сетевой айдишник, эпоху, ссылку на объект (если есть), список доменов.
	
	Эпоха увеличивается после каждого удаления объекта. Разница эпох позволяет отловить передачу данных для новой/старой версии объекта.
	Переполнение эпохи допустимо.

	Если отправлять конструкторы перед другими доменами, то несовпадение эпохи может быть: 
	* признаком переупорядовачивания сообщений,
	* неполученным (неотправленным) деструктором - можно игнорировать, если получен конструктор с эпохой на единицу больше,
	* багом сетёвки. 

- Деление сетевых объектов на домены
	-> Сетевые объекты раскладываются на сетевые домены - части, которые передаются по сети самостоятельно.
	Сетевой домен обрабатывает отправку и получение данных (если есть),
	указывает получателей (для SV->CL) и допустимых отправителей (для CL->SV).
	Сетевой домен отвечает за переотправку своих данных новым игрокам.

- Инициализация объекта
	-> Объект имеет домен переменной 0 (конструктор/деструктор).
	Когда передаются пустые данные, это деструктор, когда непустые - конструктор.

	На сервере после инициализации объекта (или после вызова функции :Net_Initialized() для статических объектов) включается домен 0 с данными для конструктора. Данные включают в себя тип объекта и пользовательские параметры конструктора.
	Домен 0 отправляет данные, кому надо.

	Клиенты получают сообщение для домена 0 для слота с объектом или без него. Если размер данных != 0, это конструктор.
	Если слот имеет объект, объект удаляется.
	Из сообщения читается тип объекта, создаётся неинициализированный объект этого типа.
	Вызывается :Net_Setup(), что создаёт домены.
	Домен 0 читает данные.
	Вызывается :Init() и :Net_Initialized().

	- Порядок инициализации на клиенте
		-> Инициализировать в порядке получения сообщений конструктора. Зависимость между объектами (напр. необходимость иметь инициализированным другой объект для полной инициализации этого) решается через второю фазу инициализации и каллбек с семафором.


- Передача нормальных данных
	Данные содержатся в домене N != 0.
	Реализация домена описана ниже.

- Удаление объекта
	-> Объект имеет домен переменной 0 (конструктор/деструктор).
	Когда передаются пустые данные, это деструктор, когда непустые - конструктор.

	На сервере удаление вызывается функцией :Remove объекта или пересозданием статического объекта (ловится повторным :Net_Initialized). 
	Всем, кто знают что-либо об объекте, отправляется пустое сообщение домена 0.
	Затем обнуляется слот и эпоха увеличивается на 1.

	На клиенте удаление объекта вызывается получением пустого сообщения домена 0 или вызовом :Remove или пересозданием статического объекта.
	Случай повторного удаления статического объекта (один раз через hot reload на клиенте, один раз через сетёвку) отлавливается совпадением эпох.
	При удалении объекта обнуляется слот и эпоха увеличивается на 1.

	- Правильная обработка случаев, когда объект был удалён, был создан новый объект с тем же айдишником, 
	и затем пришли данные для старого объекта
		-> Движковая сетёвка, ВРОДЕ БЫ, не может прислать старые данные после новых.
		Т.е. единственный способ, как такое может получиться, это из-за багов моей сетёвки.
		-> Можно ожидать, что в нормальных условиях увеличение эпохи на +1..+10 - это более новый объект, а всё остальное - wraparound от более старых.
	
- Сетёвка для статических классов
	-> Объект готов к сетёвке не по завершению :OnInit, а по вызову :Net_Initialized.
	Объект сбрасывается при повторном вызове :Net_Initialized (отловит hot reload)

- Фильтрация отправки (SV->CL)
	-> Фильтр является классом с методами
	FILTER:FillRecipientList(recip: CRecipientList)
	FILTER:GetPlayers() -> array(Player)
	FILTER:AddOnChanged(cb: fn()) и FILTER:RemoveOnChanged(cb: fn()) -- каллбек вызывается при изменении списка игроков.
	
	Отправлять данные тем, кто не получил конструктор, смысла нет.
	Значит, фильтр должен исключать всех, кто не знает об объекте.

	- Композиция фильтров
		AndFilter({Filter1, Filter2, Filter3})
		OrFilter({Filter1, Filter2, Filter3})

	- Когда перепроверять фильтр отправки?
		-> Вообще, только для доменов-переменных. Для доменов-событий это не нужно.
		
		Если фильтр - глобальный, то при подключении новых игроков
		Если фильтр - по PVS/PAS, то при входе/выходе игрока в зону видимости/слышимости.
		Если фильтр - по разрешению, то при смене разрешения.

	- Учёт варианта, когда домен-переменная изменилась, пока игрок был отфильтрован, а затем игрок прошёл фильтр.
		-> Нужно отслеживать отправочную эпоху (увеличивать на 1 при каждой смене значения переменной) 
		и то, какую отправочную эпоху наблюдал игрок. Так можно узнать, что у игрока старые данные.

	- Должен ли клиент иметь возможность знать, что он отфильтрован доменом? Если да, то как?
		-> Полезно для объектов, выполняющих работу по данным с сервера.
		Но нужно это, если данные отправляются не потоком, а редко, но при этом работа есть часто.

		Другие аддоны этот вопрос игнорируют.

- Фильтрация получения (CL->SV)
	-> Фильтр является функцией fn(ply: Player) -> allowed: bool.
	Нужно исключать тех, кто не получал конструктор.
	Сверх этого, ничего особо не нужно.
	
	Для каскадирования можно сделать фильтр, вызывающий фильтр владельца.
	ООП не нужно.
	function Obj:NetSetup()

		local filter = function(ply)
			return self.OwnerObj.GlobalFilter(ply) and ply:IsAdmin()
		end
	end

- Откладывание второй фазы инициализации до доступности необходимых объектов-ссылок
	-> Фактически, я реализую каллбек с семафором (счёткиком невыполненных предусловий).
	function Obj:NetSetup()
		local asyncinit = CR.SemaphoredCallback()
		self._asyncinit = asyncinit
		
		-- [setup domains]
		
		if CLIENT then
			asyncinit:WaitFor(self._dependencySlot1)
			asyncinit:WaitFor(self._dependencySlot2)
		end
	end
	
	function Obj:OnInit(...)
		self.Base.OnInit(self, ...)
		
		self._asyncinit:DoOnce(function() self:AsyncInit() end) -- Вызывает каллбек сразу, если семафор в нуле
	end
	
	function Obj:AsyncInit()
		local dep1 = self._dependencySlot1.Obj
		local dep2 = self._dependencySlot2.Obj
		
		assert(IsValid(dep1) and IsValid(dep2))
	end
	
- Реализация домена
	- Причина отправки
		- Вызов события (для EventDomain) или обновление данных (для VarDomain)
		- Попадание новых игроков в фильтр отправки (для VarDomain и InitDomain, SV->CL)
		- Готовность объекта-владельца к отправки по сети (для VarDomain в обе стороны, для InitDomain, SV->CL)
		- Игрок с устаревшими данными вновь попадает в фильтр отправки (для VarDomain, SV->CL) (доп. фильтрация по игрокам с устаревшими данными)
		- Отправка последующих частей домена с несколькими частями (для MultipartEventDomain, MultipartVarDomain)
	- Логика отправки
		- EventDomain (SV->CL, CL->SV): отправить сообщение (userdata) получателям
		- InitDomain (SV->CL): отправить сообщение (typename:string, userdata) получателям с Recieved[ply] ~= true, установить Recieved[ply] = true
		- VarDomain (SV->CL): отправить сообщение (userdata) получателям с RecievedVersion[ply] ~= CurVersion, установить RecievedVersion[ply] = CurVersion 
		- VarDomain (CL->SV): отправить сообщение (userdata) серверу
		- MultipartEventDomain (SV->CL, CL->SV):
			1. Запомнить список получателей, данные. Расчитать число частей, если не константа.
			2. Для каждой части отправить сообщение получателям.
				- Формат SV->CL: (part: uint, last_part: bool, data[part]: userdata)
				- Формат CL->SV: (part: uint, part_count: uint, data[part]: userdata)
				- Формат в обе стороны, если число частей константно: (part: uint, data[part]: userdata)
				- При запуске нового события, в зависимости от конфигурации домена, либо остановить это и начать новое, либо продолжить это и пропустить новое.
		- MultipartVarDomain (SV->CL)
			1. Запомнить версию, данные. Расчитать число частей, если не константа.
			2. Запомнить список получателей - игроков по фильтру с RecievedVersion[ply] ~= CurVersion 
			3. В цикле отправлять части по порядку получателям.
				- (Если игрок ушёл из списка получателей, всё-равно доотправить все части)
				- (Если в списке получателей появились новые игроки, они будут обработаны при следующем запуске логики отправки)
				- Формат SV->CL: (part: uint, last_part: bool, data[part]: userdata)
				- Формат, если число частей константно: (part: uint, data[part]: userdata)
				- При изменении данных остановить цикл, начать отправку заново.
			4. Установить RecievedVersion[ply] = CurVersion
		- MultipartVarDomain (CL->SV)
			1. Запомнить данные. Расчитать число частей, если не константа.
			2. В цикле отправлять данные по порядку серверу
				- Формат: (part: uint, part_count: uint, data[part]: userdata)
				- Формат, если число частей константно: (part: uint, data[part]: userdata)
				- При изменении данных остановить цикл, начать отправку заново.
	- Низкоуровневоая отправка сообщения
		- Начать сетевой пакет
		- Отправить (net_id, generation, domain_id)
		- Отправить пользовательские данные
		- SV->CL: отправить по списку получателей
		- CL->SV: отправить серверу
	- Низкоуровневое получение сообщения
		- Прочитать (net_id, generation, domain_id)
		- Если нету активного слота с (net_id, generation)
			- CL->SV: проигнорировать сообщение (кривые данные или хацкер на клиенте)
			- SV->CL:
				-> domain_id != 0 -> выдать ошибку (клиент получил данные для устаревшей/будущей версии объекта, игнорируем)
				-> domain_id == 0, есть дополнительные данные -> конструктор
					-> слота с таким net_id нету вообще -> создать пустой слот
					-> слот есть, generation не совпадает -> сбросить слот
				-> domain_id == 0, нет дополнительных данных -> особый случай деструктора
					-> слота с таким net_id нету вообще -> проигнорировать сообщение (объекта *уже* нету)
					-> слот есть, generation не совпадает -> продолжить обработку
		- Если domain_id == 0 и допополнительных данных нет (деструктор):
			-  Сбросить слот, задать поколение generation + 1
		- Если domain_id == 0 и есть дополнительные данные (конструктор):
			- Прочитать (typename: string)
			- Создать в слоте объект типа typename
			- Задать слоту поколение generation
			- Создать init-домен для объекта, вызвать его обработчик.
			- Вызвать инициализатор объекта
		- Если domain_id != 0
			- Получить домен из слота
			- Вызвать обработчик домена
	- Логика получения
		- EventDomain (SV->CL, CL->SV): проверить фильтр отправителя (если CL->SV), прочитать сообщение (userdata) в переменные
		- InitDomain (SV->CL): прочитать сообщение (userdata) в переменные
		- VarDomain (SV->CL, CL->SV): проверить фильтр отправителя (если CL->SV), прочитать сообщение (userdata) в переменные
		- MultipartEventDomain, MultipartVarDomain (SV->CL):
		 	1. Прочитать (part: uint, last_part: bool)
			2. Если cur_part == nil и part == 0:
				-> Вызвать обработчик начала передачи
				-> cur_part = 0
			3. Если (cur_part == nil or cur_part != part):
				- Выдать ошибку (сервер отсылает части в неправильном порядке)
				(Автоматически будет выдавать ошибки при получении следующих частей, до завершения всего фрагментированного сообщения)
			4. Вызвать обработчик части - прочитать (data[part]: userdata)
			5. Если last_part:
				-> Вызвать обработчик конца передачи
				-> cur_part = nil
			6. Если не last_part
				-> cur_part = cur_part + 1
		- MultipartEventDomain, MultipartVarDomain (CL->SV):
			???


## Сетёвка: использование

local AdminFilter = CR.Net.PermissionFilter("admin") -- Example custom filter

local Obj = CR.Class.Register("Test.Obj")
CR.Class.MakeConstructableRemovable(Obj)

function Obj:Net_Setup() -- Called by the net lib
	self.Net_SendFilter = CR.Net.PVSFilter()
	
	-- Transmitted from server to all clients allowed by the recieve filter.
	-- Transmitted on update (:MarkUpdated()) or to newly-connected allowed players.
	self._varDomain = self:Net_AddDomain(CR.Net.VarDomain({
		Send = function(domain) -- SERVER
			net.WriteVector(self.Pos)
			net.WriteUInt(self.Var1, 32)
			net.WriteString(self.Var2)
			net.WriteUInt(#self.Array, 16)
			for _, v in ipairs(self.Array) do
				net.WriteUInt(v.Num, 4)
				net.WriteVector(v.Pos)
			end
		end,
		Recv = function(domain, len) -- CLIENT
			self.Pos = net.ReadVector()
			self.Var1 = net.ReadUInt(32)
			self.Var2 = net.ReadString()
			
			local arr = {}
			for i = 1, net.ReadUInt(16) do
				arr[i] = {
					Num = net.ReadUInt(4),
					Pos = net.ReadVector()
				}
			end
			
			self.Array = arr

			self:OnVarsUpdated()
		end
	})) -- Considered updated on self._varDomain:MarkUpdated() or on object init.

	self._eventDomain = self:Net_AddDomain(CR.Net.EventDomain({
		-- You don't have to use extra parameters (they are varargs BTW), you can also read fields from the object.
		Send = function(domain, msg) -- SERVER
			net.WriteString(msg)
		end,

		Recv = function(domain, len) -- CLIENT
			self:OnTrolled(net.ReadString())
		end,

		SendFilter = AdminFilter, -- ANDed with self.Net_SendFilter
	}))
end

function Obj:Net_SendInit() -- Send constructor parameters, basically
	net.WriteString(self.Name)
end

function Obj:Net_RecvInit(len) -- Receive constructor parameters. Called before :Init on client.
	self.Name = net.ReadString()
end

function Obj:Init(name)
	if SERVER then
		self.Name = name
	else
		-- On client, self.Name was already set by :Net_RecvInit
	end
	
	print("My name is", self.Name)
end -- :Net_SendInit may be called as soon as after :Init finished

if SERVER then
	function Obj:SetPos(pos)
		self.Pos = pos
		self:PosUpdated()
	end

	function Obj:PosUpdated()
		self.Net_SendFilter:SetPos(self.Pos)
		self._varDomain:MarkUpdated()
	end

	function Obj:Change()
		self.Var1 = self.Var1 + 1
		self._varDomain:MarkUpdated()
	end
	
	function Obj:DoSomeTrolling(who)
		self._eventDomain:Send("Trolled by "..who:Nick())
	end
else
	function Obj:OnTrolled(msg)
		-- Only sent to admins who see Obj
		chat.AddText(self.Name..":"..msg)
	end

	function Obj:OnVarsUpdated()
		print("New Var2:", self.Var2)
	end
end

concommand.Add("create_object", function(ply, _, args) -- SERVER
	if IsValid(ply) and IsValid(ply.TheObject) then
		print("Already created!")
		return
	end 
	
	local name = args[1]

	local obj = Obj:New(name) -- Creates the object. It will be networked as soon as possible.
	if IsValid(ply) then
		obj:SetPos(ply:EyePos())
	end
end)

concommand.Add("remove_object", function(ply) -- SERVER
	if IsValid(ply) and IsValid(ply.TheObject) then
		ply.TheObject:Delete() -- Deletes the object. Removal will be networked to clients who know about the object.
		ply.TheObject = nil
	end
end)
